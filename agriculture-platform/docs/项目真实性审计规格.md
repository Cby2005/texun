# 温室草莓智慧农业平台真实性审计规格

## 目标

验证项目是否围绕温室草莓形成可运行、可追踪、可解释的业务闭环，而不是仅有页面、演示数据或功能名称。审计对象包括平台管理员、草莓基地管理员、溯源企业用户、农技专家、普通农户、内容审核员和消费者的核心操作。

## 技术栈

- 后端：Java 17、Spring Boot 3.2.5、Spring Security、MyBatis-Plus、MySQL 8
- 前端：Vue 3、Vite 5、Pinia、Element Plus、ECharts
- 模型：Python FastAPI、Ultralytics YOLO、本地 `model/best.pt`
- 外部依赖：高德天气、可选 Roboflow、预留 Agent 诊断地址

## 命令

- 后端构建：`cd backend && mvn test`
- 前端构建：`cd frontend && npm run build`
- 前端开发：`cd frontend && npm run dev -- --host 127.0.0.1 --port 5173`
- 后端开发：`cd backend && mvn spring-boot:run`
- YOLO 服务：`powershell -ExecutionPolicy Bypass -File yolo-service/start.ps1`

## 项目结构

- `backend/src/main/java/com/agriculture`：业务接口、认证、持久化和外部模型调用
- `frontend/src`：路由、页面、状态和后端请求适配
- `yolo-service`：本地草莓病虫害模型适配器
- `model`：模型权重
- `sql`：初始化和升级脚本
- `dataset`：草莓产量数据源
- `docs`：系统设计和审计文档

## 代码风格

接口返回统一使用 `Result<T>`；前端请求统一经过 `src/utils/request.js`；业务状态使用明确枚举值，禁止用随机数或固定文本冒充模型结果。外部模型、数据库和页面之间的字段名称及状态含义必须一致。

## 测试策略

1. 静态检查路由、前端请求、后端接口、数据库表和模型适配器的完整调用链。
2. 使用真实账号执行登录、生产管理、溯源查询和智能诊断的 HTTP 集成测试。
3. 使用真实草莓图片验证 `best.pt` 推理结果，不以模拟返回代替。
4. 在浏览器检查核心页面、网络请求、控制台和可访问名称。
5. 后端和前端构建必须通过；关键业务应具备自动化测试。

## 约束

- 始终：区分真实能力、规则算法、演示数据和模拟功能；结论必须附文件或运行证据。
- 先确认：数据库破坏性迁移、依赖升级、批量删除现有功能。
- 禁止：提交密钥；把随机结果描述成 AI；用静态页面替代业务闭环；在消费者公开接口暴露管理数据。

## 成功标准

- 生产端：农场、地块、作物、设备、环境、任务和产量数据能够关联到同一草莓生产上下文。
- 溯源端：批次 `B202406001` 能串联生产、加工、质检、仓储、物流、销售并公开查询。
- 诊断端：上传图片后实际调用 `best.pt`，保存识别结果，并对 Agent 不可用或低置信度提供明确降级状态。
- 知识端：文章、视频、RAG 和推荐的数据来源、算法性质及审核状态真实可解释。
- 权限端：各角色只能访问其职责内接口；公开消费者入口无需登录且不泄露敏感字段。
- 工程端：不存在阻断构建的错误、硬编码密钥、核心随机模拟、数据库结构漂移或无上限查询。

## 待确认问题

- Agent 诊断计划接入哪一个真实模型或服务。
- 无人机巡检识别是复用草莓 YOLO，还是单独训练航拍模型。
- 产量预测定位为统计趋势工具，还是需要后端机器学习模型。
- 内容审核员角色是否继续保留，以及对应账号和权限范围。
